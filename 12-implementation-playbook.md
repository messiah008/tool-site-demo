# 实施流程总纲 (Implementation Playbook)

> **版本**: v1.0
> **最后更新**: 2026-07-02
> **状态**: 顶层执行指南,所有实施以此为准
> **目的**: 保障系统最终落地 + 支持持续迭代 + AI 跨环境一致性

---

## 一、文档定位

本文件是项目的**顶层执行指南**,解决三个问题:

1. **怎么落地** — 从现在到上线的完整流程,分阶段、有验收
2. **怎么迭代** — 加工具/改工具的标准流程,可重复执行
3. **怎么保持一致** — AI 跨环境接手时的规范,避免漂移

**所有开发者(人和 AI)开始工作前必读本文件**。

---

## 二、核心原则:三律一循

### 2.1 三律(必须遵守)

| 律 | 内容 | 违反后果 |
|----|------|---------|
| **律一:Schema 优先** | 所有变更从 schema 开始,代码生成,不手写 | CI 失败,一致性破坏 |
| **律二:文档同步** | 代码变更必须同步更新对应文档和进度记录 | 接手者困惑,返工 |
| **律三:小步快跑** | 每个阶段有明确验收,过不了不进下一阶段 | 风险累积,返工成本高 |

### 2.2 一循(循环改进)

```
        ┌──────────────────────────┐
        │  1. 识别需求              │
        │  (新工具/改工具/修 Bug)   │
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  2. 改 Schema(SSOT)       │
        │  (registry/tokens/openapi)│
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  3. 生成代码              │
        │  (build:tokens/gen:api)  │
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  4. 实现业务逻辑          │
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  5. 本地校验 + 测试       │
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  6. 更新进度文档          │
        │  (01-progress.md)        │
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  7. CI 校验 + 提交        │
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │  8. 上线 + 监控           │
        └────────────┬─────────────┘
                     │
                     └──→ 回到 1(新需求)
```

**每个变更都走这 8 步**,无论是加一个工具还是改一个颜色。

## 四、持续迭代流程(加工具/改工具)

系统上线后,持续迭代的标准流程:

### 4.1 加新工具的流程

**触发**: 用户反馈/选品发现新需求

**步骤**:

```
1. 评估选品
   - 搜索量?(百度指数/微信指数)
   - 竞争度?(同类工具多少)
   - 技术可行性?(纯前端/需后端)
   - 付费潜力?(免费引流 or 付费工具)

2. 改 Schema(SSOT)
   - 编辑 schemas/tools.registry.json,添加工具定义
   - 用 ajv 校验: ajv validate -s tool.schema.json -d tools.registry.json

3. 选 backend
   - 纯前端工具(JSON/Base64 等): backend="frontend",无需后端
   - PDF 处理: backend="stirling",接入 Stirling-PDF
   - 高质量转换: backend="wps-worker",走 WPS
   - 新类型: 实现新 Worker(见 10-worker-integration.md 4.2)

4. 实现前端
   - 复制 prototypes/pdf-to-word.html 模板
   - 改工具特定 UI
   - 放到 frontend/src/tools/{id}/index.vue

5. 实现/对接后端
   - 复用已有 backend: 无需新代码
   - 新 backend: 按 Worker 模板实现

6. 本地测试
   - 启动前端,首页自动出现新卡片
   - 上传测试文件,验证全流程
   - 检查错误码符合 E001-E099

7. 更新进度
   - 在 01-progress.md 记录新增工具
   - 更新 README 文档导航(如需)

8. CI 校验 + 提交
   - schema 校验通过
   - 前端构建通过
   - E2E 测试通过

9. 上线 + 监控
   - 灰度发布
   - 观察 24 小时数据
   - 根据反馈迭代
```

**时间预算**:
- 纯前端工具: 0.5-1 天
- Stirling 后端工具: 1-2 天
- WPS 后端工具: 2-3 天
- 新 Worker 类型: 5-7 天

### 4.2 改工具的流程

**触发**: Bug 修复/功能增强/UI 优化

**步骤**:

```
1. 定位变更范围
   - 只改前端? → 改 Vue 组件
   - 改接口? → 先改 openapi.yaml,再 gen:api
   - 改视觉? → 先改 design-tokens.json,再 build:tokens
   - 改工具元数据? → 改 tools.registry.json

2. 改 Schema(如需)
   - 修改对应 SSOT 文件
   - 执行生成命令

3. 改业务代码

4. 测试
   - 单元测试
   - E2E 测试
   - 视觉走查(如改视觉)

5. 更新进度
   - 01-progress.md 记录变更

6. CI + 提交
```

### 4.3 迭代节奏

| 周期 | 动作 |
|------|------|
| 每周 | 新增 2-3 个工具(从选品清单) |
| 每两周 | 优化 1-2 个现有工具(根据数据) |
| 每月 | 复盘,更新选品清单 |
| 每季度 | 战略复盘,调整方向 |

## 五、AI 跨环境一致性保障

### 5.1 自包含原则

`tool-site-project/` 目录是**自包含**的:
- 复制到任何环境都能用
- AI 读 README.md 即可接手
- 所有真相在 schemas/ 下,不依赖外部

### 5.2 AI 接手流程

新环境 AI 接手时:

```
1. 读 README.md → 了解项目结构和 schema 位置
2. 读 01-progress.md → 了解当前进度和阶段
3. 读 07-consistency-link.md → 理解一致性机制
4. 读 08-ai-collaboration.md → 了解工作规范
5. 读本文件(12-implementation-playbook.md) → 了解实施流程
6. 读 CLAUDE.md → 了解项目记忆
7. 接到任务 → 按本文件第四章流程执行
```

### 5.3 AI 协作规范

| 场景 | 规范 |
|------|------|
| AI A 加工具 | 改 registry,提交,AI B 接手时读 progress 知道 |
| AI A 改 API | 改 openapi.yaml,gen:api,提交,AI B 拉取自动同步 |
| AI A 改视觉 | 改 tokens,build:tokens,提交,所有页面同步 |
| 多 AI 并行 | 各改各的 schema,git 合并,CI 校验 |
| AI 跨环境 | 复制目录,新 AI 读 README 接手,改完 git 同步 |

### 5.4 防 AI 漂移机制

AI 容易"自由发挥",以下机制防止漂移:

| 机制 | 作用 |
|------|------|
| JSON Schema 校验 | AI 改 registry 必须符合 schema,否则 CI 失败 |
| OpenAPI 比对 | AI 改后端必须匹配契约,否则 CI 失败 |
| 硬编码色值检查 | AI 不能写死颜色,必须用变量 |
| 进度记录强制 | AI 改完必须更新 01-progress.md |
| CLAUDE.md 引用 | AI 每次启动读规范 |

## 六、文档维护规范

### 6.1 文档分类

| 类型 | 文件 | 更新时机 |
|------|------|---------|
| **SSOT(真相源)** | schemas/* | 代码变更前先改 |
| **进度记录** | 01-progress.md | 每次变更后更新 |
| **设计文档** | 02-11.md | 架构变更时更新 |
| **执行指南** | 本文件(12) | 流程变更时更新 |
| **AI 规范** | CLAUDE.md, 08 | 协作方式变更时更新 |

### 6.2 文档更新检查清单

每次代码变更,确认:

- [ ] 改了 SSOT 吗?(schemas/)
- [ ] 执行了生成命令吗?
- [ ] 更新了 01-progress.md 吗?
- [ ] 相关设计文档需要更新吗?
- [ ] CLAUDE.md 需要更新吗?

### 6.3 文档版本约定

- 每个文档顶部有 `版本` 和 `最后更新`
- 重大变更 bump major(v1 → v2)
- 小变更 bump minor(v1.0 → v1.1)
- 变更记录在 01-progress.md

## 七、风险管理

### 7.1 关键风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| WPS License 谈不下来 | 中 | 高 | 备选 PaddleOCR |
| 闲鱼封号 | 中 | 中 | 多账号+私域 |
| 流量起不来 | 中 | 高 | 多渠道引流 |
| WPS CLI 变更 | 低 | 高 | 错误码降级 |
| Redis 故障 | 低 | 高 | RDB+哨兵 |

### 7.2 决策点

每个阶段验收门禁是 **Go/No-Go** 决策点:
- 通过 → 进下一阶段
- 不通过 → 排查问题,延期或调整

**禁止跳过门禁进下一阶段**(律三:小步快跑)。
